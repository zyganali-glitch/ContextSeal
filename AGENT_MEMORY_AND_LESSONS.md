# ContextSeal Lessons Learned

Every incoming agent should read this file before touching product claims, demo surfaces, or DataHub proof logic.

## Current minefield

1. The strongest differentiator is the passport and evidence-bound approval artifact, not generic impact analysis.
2. Current live DataHub proof captures raw MCP read/write evidence, but the normalized impact path shown in the product is still fixture-derived until a target-derived live graph contract exists.
3. The code currently traverses asset-level lineage; if field-level precision is not truly implemented, claims must be downgraded rather than stretched.
4. Reviewer and judge trust is more fragile than feature count. One overclaim costs more than one missing stretch feature.
5. README and Devpost tone were too disclaimer-heavy. Future edits should lead with strengths, then state boundaries.
6. Zero paid new tooling is a hard execution constraint for the current upgrade roadmap. Prefer local, OSS, or free tiers only.
7. The best agentic addition is an explanatory layer that helps operators act; it must not become decision authority.
8. For this hackathon, AI cannot remain a sidecar. It must be visible in the winning demo path within the first minute.
