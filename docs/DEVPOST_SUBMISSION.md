## Project name

ContextSeal

## Tagline

Every data change ships with proof, not confidence.

## Challenge category

**Primary:** Agents That Do Real Work

**Secondary fit:** Metadata-Aware Code Generation & Development

## Inspiration

AI coding agents can generate valid SQL and dbt code while missing the organizational context that makes a change safe. A repository does not reveal that one field feeds a distant dashboard, crosses team ownership, carries a privacy term, or sits on a model-scoring path. DataHub already knows those relationships. We built ContextSeal to turn that context into a decision boundary before a risky data change ships.

## What it does

ContextSeal accepts a proposed column rename, drop, or type change. It uses DataHub MCP to retrieve exact target governance metadata, exhaust the target schema through `list_schema_fields`, discover bounded downstream lineage, obtain exact asset-to-asset paths, and preserve available dataset-query records. A deterministic policy engine reconstructs the blast radius and explains every risk point.

Instead of generating a destructive operation, the agent produces a reviewable expand–migrate–contract package: a dbt model, schema tests, a separately named rollback, and an impacted-owner brief. A human approves or rejects only that exact safe scope.

Approval creates an expiring SHA-256 change passport binding the request, policy, raw and normalized DataHub evidence, impact, risk, every generated byte, and the complete human decision. When the independent runtime mutation gate is enabled, ContextSeal writes certification properties, an appended description, and a decision document to DataHub, then reads back structured fields and exact document bindings.

## Why this is an agent

ContextSeal is not a chatbot or metadata summary. Given a change intent, it autonomously selects bounded DataHub tools, follows downstream targets, constructs exact paths, makes a deterministic decision, generates working artifacts, transitions through persistent states, pauses at a human authority boundary, performs an approved external action, and verifies the outcome.

Its loop is:

**read context → decide → generate safe work → approve/reject → passport → write back → verify**

## What real work it performs

- Reconstructs downstream impact paths instead of returning a flat search list
- Converts governance, criticality, lineage, quality, incident, and query evidence into named policy findings
- Replaces direct destructive requests with reversible dbt delivery artifacts
- Produces an owner handoff that preserves path and accountability context
- Binds approval to one exact expiring manifest
- Executes three bounded DataHub metadata mutations only after all gates pass
- Separately verifies durable target fields and decision-document bindings

It deliberately does not execute warehouse SQL or merge code automatically.

## How we built it

- Dependency-light Node.js 20+ deterministic core and HTTP API
- Official `mcp-server-datahub` v0.6.0 launcher package through local stdio, plus streamable-HTTP support (the captured handshake separately reports protocol `2025-03-26` and `serverInfo` `datahub`/`3.4.4`)
- Raw-to-normalized evidence binding with complete schema pagination, truncation, inspectability, and count checks
- Exact path call for every discovered downstream asset, with path lengths and reconstructed impact constrained by the same policy `maxHops`
- Versioned JSON risk policy and typed contracts
- dbt artifact generator with identifier/type safety and destination-collision checks
- Passport v2 SHA-256 manifest
- Required operator bearer token, exact live-target allowlist, approval gate, runtime mutation gate, and replay protection
- Bounded retry for read-only post-write verification; mutation calls are never retried
- No-framework accessible dashboard with explicit hosted, fixture, and live modes
- Node 20/24 CI, Docker fixture smoke, deterministic demo generation, server smoke, and offline committed-evidence validation

## How it uses DataHub

DataHub is the system of context **and** the durable memory layer.

Read and path tools:

- `get_entities` — exact target identity and governance metadata
- `list_schema_fields` — complete unfiltered target-schema pagination for source and destination validation
- `get_lineage` — bounded downstream discovery
- `get_lineage_paths_between` — endpoint-bound path for every discovered target
- `get_dataset_queries` — inspectable query records when available

Approved mutation tools:

- `add_structured_properties`
- `update_description` in append mode
- `save_document`

Verification tools:

- `get_entities` — read back status, risk score, passport ID, validity, and description reference
- `grep_documents` — verify passport ID, manifest hash, and target URN inside the exact returned document

The committed live-local proof uses a disposable DataHub Core catalog containing only ContextSeal synthetic metadata. It has native Dataset, DataJob, and Dashboard lineage. The query tool returned zero records in that catalog, and ContextSeal preserved zero instead of inventing usage. The fixture separately demonstrates two clearly labeled synthetic query signals.

The current committed run records 10 MCP reads, a complete three-field target schema, six downstream assets (2 Datasets, 2 DataJobs, 2 Dashboards), six exact paths, and zero query records. The direct request remained deterministically `70 / BLOCKED`; only the generated safe scope was approved. Exactly three bounded mutations returned `PASS`, followed by a separate durable read-back `PASS`.

## What makes it original

Most catalog agents help a user find or describe metadata. ContextSeal changes what an engineer or coding agent is allowed to do. Its output is not merely an answer: it is a reversible delivery bundle plus an expiring, graph-backed certification that the next human or agent can inspect in DataHub.

The direct request may be `BLOCKED` while the generated non-destructive alternative remains eligible for human approval. Keeping those two decisions separate is the core product idea.

## Challenges we ran into

**Protocol success is not business success.** MCP can deliver a tool result whose `isError` flag is true. We made tool-level errors fail closed and preserved partial receipts.

**Lineage counts are not paths.** We added one exact path request for every discovered target and reject truncated or inconsistent results.

**An entity preview is not a complete schema.** `get_entities` may truncate large schemas, so field existence and destination absence now come only from continuous, count-checked `list_schema_fields` pages.

**Evidence can be internally self-consistent and still be unrelated.** The validator now reconstructs normalized context, impact, risk, and artifacts from the bound raw calls and policy.

**DataHub is eventually consistent.** We never retry mutations. We retry only read-only verification within a short bounded deadline and preserve each attempt.

**Honesty can improve the demo.** Our local query tool returned zero records. Showing that zero—and keeping fixture query examples visibly synthetic—made the product more credible.

## Accomplishments we are proud of

- A complete read–decide–act–write-back–verify DataHub loop
- Explainable branching impact paths with full path provenance
- Safe, merge-reviewable dbt artifacts rather than destructive SQL
- A passport that invalidates on request, policy, context, evidence, risk, impact, artifact, approval, or expiry drift
- Fail-closed live API and mutation controls
- A professional dashboard where `PASS`, `WARN`, `FAIL`, `NOT_RUN`, `STALE`, and `FIXTURE` never blur together
- Reproducible offline fixture and committed synthetic-local DataHub proof
- A generic executable schema-change certification skill prepared for upstream review

## What we learned

Context is most valuable when it changes an action, not when it only improves an answer. DataHub makes it possible to move agent safety out of prompt instructions and into a repeatable workflow grounded in the organization's graph.

We also learned that `NOT_RUN` is a feature. A prepared payload, generated SQL file, mutation receipt, and durable read-back are different claims. Keeping them separate makes an agent easier to trust.

## Honest limitations

ContextSeal is a hackathon prototype. It does not execute warehouse SQL, merge or deploy code, prove that DataHub contains complete lineage, guarantee security, or claim production/customer use. The public hosted page is a generated fixture walkthrough. The committed DataHub evidence came from a disposable local instance with synthetic metadata. Full saved-document byte retrieval was unavailable in that local MCP surface, so the proof verifies exact returned-document bindings rather than claiming full-body equality.

## Data used

Only synthetic retail metadata owned by this repository. ContextSeal reads metadata, never source data rows. No customer or production data is included.
